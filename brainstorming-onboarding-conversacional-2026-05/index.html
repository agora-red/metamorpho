10 / 16
ARQUITECTURA TRANSVERSAL

Lo que tiene que ser cierto
en cualquiera de los pasos.

01

Surface mixta

Chat principal + sidebar visual que cambia según el paso: calendario, página pública, equipo, card de servicio.

02

Priors por categoría

Antes del primer mensaje, Claude ya sabe los defaults razonables (peluquerías: turnos cada 30, corte 45, lun-sáb). Vendor solo conversa diferencias.

03

Mismo data model

Las tool calls escriben los mismos endpoints/repos que la wizard actual. Sin migración de schema. Sin riesgo de divergencia.

04

Save state continuo

Cada respuesta persiste server-side. Vendor puede abandonar y volver — el chat retoma exactamente donde quedó.

05

Skippable inteligente

"Saltame esto, lo configuro después" en cada paso. El sistema sabe qué es crítico (horarios + primer servicio) y qué no.

06

Formato mixto

No todo es texto libre. Teléfono, plan, MP OAuth, privacy: componentes estructurados embebidos en el chat. El agente elige el formato que más conviene al dato.

11 / 16
RIESGOS Y TRADEOFFS

Lo que puede salir mal.

Si elegimos este camino, estos son los puntos donde hay que tener atención explícita o aceptar el costo.

Costo ingenieril del live preview

El calendar debe renderizar con state parcial e inconsistente durante toda la conversación. No trivial. Si el preview falla, el chat solo es peor que la wizard.

Spike de 1 semana antes de comprometernos al approach completo.

Defaults silenciosos del agente

Riesgo de que el vendor termine con config que no entiende porque Claude asumió cosas sin verbalizarlas.

Cada tool call con default debe decirlo: "asumí turnos cada 30, ¿está bien?".

No todo gana con chat

Pasos triviales (phone, plan con un click) no son más rápidos en conversación que un input. Forzar chat es ideología.

Regla: el agente elige el formato según el dato. Chat conduce, no monopoliza.

Compliance steps siguen siendo UI

Privacy policy, T&Cs, OAuth con MP: no se conversan. Hay que diseñar bien la transición chat → UI estructurada → chat.

Componentes embebidos en el chat (no modals). El chat espera.

Branching plan/rol vive en el agente

La lógica condicional pasa de routing explícito a prompt del agente. Más fácil iterar; más difícil auditar y testear.

Tests de regresión por flow + observabilidad de qué path tomó cada vendor.

Tamaño del proyecto

Replantear todo el funnel de signup es serio. ¿Hacemos todo o solo Setup 03?

MVP: reemplazar SOLO Working Hours por chat. Si el preview funciona y el lift es real, expandir.

PARA DISCUTIR · ACÁ EMPIEZA LA SESIÓN

6 preguntas abiertas.

No tenemos respuesta a ninguna. Si la tuviéramos, no estaríamos haciendo brainstorming. La salida del meeting es un sí/no por pregunta + un owner por la pregunta donde decidamos avanzar.

Q · 01 ¿Toda la wizard o solo Setup 03? Working Hours es el punto de más dolor y más ganancia. Empezar ahí reduce riesgo y demuestra el approach. ¿O hacemos todo de una para no fragmentar?
Q · 02 Build cost del live preview. El calendar con state parcial es el riesgo técnico real. ¿Hacemos un spike de 1 semana para validar feasibility antes de comprometernos al approach?
Q · 03 ¿Cómo medimos éxito? Propuesta: completion del setup completo (con frequency), time-to-first-booking, % de vendors que tocan agenda los primeros 7 días (proxy de "salí roto"). ¿Suma alguna?
Q · 04 ¿Vendors existentes también? ¿Ofrecemos "reconfigurar tu setup con AI" como flow paralelo para vendors que ya pasaron por la wizard y tienen config a medias?
Q · 05 ¿Lógica en prompt o en routing? El branching plan/rol vive en la lógica del agente (más iterable) o sigue en routing explícito (más auditable). Tradeoff real.
Q · 06 ¿Primera categoría piloto? Peluquerías parecen ideales — schedule complejo, alta densidad de vendors, priors bien entendidos. ¿O preferimos arrancar por algo más simple para reducir variables?
